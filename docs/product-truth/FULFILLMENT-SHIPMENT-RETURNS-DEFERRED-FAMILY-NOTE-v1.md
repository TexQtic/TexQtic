## 3. What This Deferred Family Means Now

Within current planning truth, `Fulfillment / Shipment / Returns` means the later merchant- or
operator-side post-order execution family beyond the currently bounded order continuity already
owned by `Orders / Checkout / Post-Purchase`.

At minimum, this deferred family covers:

- shipment preparation, dispatch readiness, and after-order execution depth beyond current bounded
  order-status visibility
- delivery exception, receipt exception, or reverse-flow continuity once those loops become
  materially supported
- returns, replacement, or broader after-order operational loops not yet evidenced as a clean
  cross-mode family today
- merchant/operator execution responsibility distinct from external logistics-service-provider
  participation

## 4. What Remains Outside This Deferred Family Note

The following remain outside this note and must not be silently absorbed into it:

- current bounded order creation, order visibility, and post-purchase continuity already owned by
  `Orders / Checkout / Post-Purchase`
- `Logistics / Custody Handoff / Service-Provider Participation` as the separate top-level family
  for custody boundaries and external logistics actors
- disputes, settlement, finance supervision, or messaging-family ownership
- warehouse, inventory, OMS/WMS, or broader implementation-program redesign

## 5. Current Launch Posture

The controlling current launch posture is:

- this family remains explicitly deferred / post-launch
- launch must keep fulfillment, shipment, and returns out of the core promise unless a bounded
  operating loop is separately defined first
- current repo truth supports post-order progression better than shipment or returns depth, which
  is not enough to claim this family materially complete

## 6. Re-entry Conditions For Later Bounded Work

This deferred family should re-enter active planning only if one of the following becomes true:

- launch scope or public/commercial promise expands to include shipment, delivery, returns, or
  after-order execution depth as a material capability
- a later bounded unit needs merchant-side fulfillment logic beyond current order-status continuity
- logistics-family work requires a clear merchant/operator-side counterpart at the handoff boundary
- repo truth later evidences a multi-mode shipment or returns runtime strong enough to justify a
  first-class family design
