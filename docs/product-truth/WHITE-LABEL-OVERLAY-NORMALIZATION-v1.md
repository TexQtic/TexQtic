## 1. Purpose and Authority

This artifact is the canonical current family-level normalization anchor for `White-Label Overlay`.

Its purpose is narrow:

- fix the lawful white-label classification under the adopted canonical model
- prevent white-label from drifting back into separate-mode, separate-pillar, or pseudo-SaaS-
  primary framing
- make adjacent-family boundaries explicit enough that later family units can inherit them without
  re-litigating white-label identity

This artifact does not authorize implementation, does not redesign adjacent families, and does not
replace Layer 0 governance truth.

## 2. Current Controlling Posture

Current repo authority already fixes the following truths:

- TexQtic is the operating system for trusted textile supply chains.
- The only governed commercial access models are B2B Exchange, B2C Tenant-Branded Commerce, and
  Aggregator Directory Discovery + Intent Handoff.
- White-label is an overlay capability and deployment/experience model, not a fourth pillar and
  not a separate commercial access model.
- Enterprise is depth within B2B, not a separate platform mode.
- Control-plane, superadmin, and WL-admin surfaces are governance/operations surfaces, not
  commercial pillars.
- Current repo truth already evidences real white-label storefront/runtime and brand-operator admin
  surfaces in bounded form.

The remaining drift is classification drift. White-label still appears in older planning language
as if it were an operating mode parallel to B2B, B2C, or Aggregator, or as if WL-admin/operator
surfaces could stand in for tenant back-office, control-plane, or domain-family truth.

## 3. Lawful White-Label Classification

The controlling classification is:

- white-label is an overlay capability
- white-label is a deployment and experience model
- white-label is a branded tenant/operator presentation layer
- white-label may sit on top of B2B and/or B2C parent surfaces where that overlay is lawful

White-label is therefore not:

- a fourth board-level pillar
- a separate governed commercial access model
- a replacement for B2B Exchange Core
- a replacement for B2C Tenant-Branded Commerce
- a replacement for Aggregator Directory Discovery + Intent Handoff
- a substitute for platform control-plane or superadmin
- a synonym for the entire domain / tenant routing / brand-surface management family

## 4. What White-Label Means In TexQtic Planning Truth

Within TexQtic planning truth, `White-Label Overlay` means the branded runtime, operator, and
deployment layer that modifies how a parent commercial access model is presented and administered
without changing the parent model's underlying classification.

At the family level, this means:

- a tenant or operator can present TexQtic capability through branded storefront or experience
  surfaces
- brand/operator-specific configuration and administration can exist as overlay-specific runtime
  continuity
- WL-admin/operator surfaces can exist as overlay back-office related to the branded experience
- deployment and brand concerns such as domain, presentation, and operator-facing configuration may
  attach to the overlay where repo truth already supports them

The correct high-level family statement is:

- White-Label Overlay is the branded presentation, operator, and deployment layer that overlays a
  parent commercial access model without replacing that parent model.

## 5. What Belongs Inside White-Label Overlay Scope

The following belong inside the WL overlay family boundary:

### 5.1 Branded runtime and storefront overlay behavior

- branded storefront or experience rendering
- white-label tenant-facing presentation differences
- overlay-specific navigation or presentation continuity tied to branded experience

### 5.2 Brand/operator-specific administration

- WL-admin or brand-operator administration directly tied to the branded storefront/runtime
- brand configuration, store profile, theme, collections, operator-facing product presentation, and
  related overlay-specific management where already evidenced in repo truth
- overlay-specific operator workflows needed to keep the branded experience coherent

### 5.3 Overlay-specific deployment concerns

- custom domain or brand-facing deployment concerns as overlay-specific needs
- brand/operator concerns that determine how a lawful parent mode is rendered or reached
- bounded domain or routing overlap only where it is specifically about the branded overlay path

## 6. What Does Not Belong Inside White-Label Overlay Scope

The following are adjacent or later families and must not be silently absorbed into WL
normalization:

### 6.1 B2B Exchange Core

- the full authenticated business exchange family
- RFQ, negotiation, order, trade, escrow, settlement, and compliance continuity as a whole B2B
  family definition
- enterprise depth classification

### 6.2 B2C Tenant-Branded Commerce

- the full B2C family definition
- the full consumer-commerce family boundary
- all public-safe browse-entry, cart, checkout, orders, and post-purchase truth as a substitute WL
  definition

### 6.3 Aggregator Directory Discovery + Intent Handoff

- directory/discovery and intent-routing as an Aggregator family definition
- any attempt to redefine Aggregator as a white-label variant

### 6.4 Tenant admin / tenant back office

- the whole tenant back-office family across memberships, org administration, integrations,
  non-overlay staff administration, and cross-mode admin controls
- generic tenant-admin truth that is not specifically overlay-owned

### 6.5 Platform control-plane / superadmin

- platform supervision, operator governance, audit, finance oversight, compliance casework,
  disputes, platform-level RBAC, or tenant-registry depth
- WL-admin interpreted as a whole control-plane or superadmin substitute

### 6.6 Domain / tenant routing / brand-surface management as a full family

- full routing, tenant resolution, brand-surface ownership, or domain-family decisions as a whole
  cross-cutting family
- any attempt to fully solve the domain/routing family through white-label normalization alone

### 6.7 Cross-mode catalog, orders, payments, identity, messaging, or governance families

- catalog/discovery cross-mode ownership
- orders / checkout / post-purchase family definition
- payments / escrow / settlement family definition
- identity / tenancy / permissions / workspace family definition
- messaging / notifications family definition
- AI governance / advisory automation or feature-governance families

## 7. Relationship To Adjacent Families

### 7.1 Relationship to B2B Exchange Core

- White-label may overlay B2B runtime or operator surfaces where lawful.
- White-label does not replace B2B as the parent commercial access model.
- Enterprise terminology must not be used to invent a white-label-adjacent separate mode.

### 7.2 Relationship to B2C Tenant-Branded Commerce

- White-label may overlay branded B2C storefront or operator surfaces where repo truth supports it.
- White-label does not become the canonical definition of the whole B2C family.
- Consumer storefront truth and broader B2C family design remain separate from WL normalization.

### 7.3 Relationship to Aggregator

- White-label does not define Aggregator and must not absorb directory or intent-handoff family
  ownership.
- If a branded overlay later exists for Aggregator-adjacent surfaces, the parent model still
  remains Aggregator.

### 7.4 Relationship to tenant back-office

- WL-admin/operator surfaces may overlap with tenant administration where they are specifically
  overlay-owned.
- Tenant back-office remains a later family and must not be collapsed into WL.

### 7.5 Relationship to control-plane / WL-admin

- WL-admin is an operator/back-office surface related to the overlay family.
- WL-admin is not itself a commercial pillar.
- WL-admin is not a synonym for control-plane or superadmin.

### 7.6 Relationship to domain / tenant routing / brand surfaces

- domain, tenant-routing, and brand-surface concerns overlap with WL because branded delivery paths
  are part of the overlay story
- that overlap does not authorize solving the full domain / tenant routing / brand-surface family
  here

## 8. Current Sources Of Drift

The main current drift classes are:

- older shorthand such as `hybrid B2B + white-label SaaS`
- older artifacts that use `white-label operating mode` language without separately fixing the
  family classification boundary
- older runtime/planning language that can make WL-admin appear co-equal with control-plane or
  tenant-admin families
- routing or tenant-type discussions that can make white-label look like an identity or canonical
  tenant type rather than a capability overlay
- local shorthand that groups enterprise and white-label together as if both were separate modes

These phrases may still describe bounded historical work, but they must not be used as current
family classification authority.

## 9. Inheritance Rules For Later Family Units

Later family-specific units must inherit all of the following:

1. White-label overlays a parent commercial access model; it does not replace it.
2. White-label is not a separate commercial access model or board-level pillar.
3. WL-admin is related to the overlay family but is not a control-plane substitute.
4. Tenant back-office work must remain separate unless a later unit proves a specific overlay-owned
   admin slice.
5. Domain / tenant routing / brand-surface work must remain a distinct later family even when it
   overlaps with WL concerns.
6. White-label work must not silently absorb B2B, B2C, Aggregator, catalog, orders, payments,
   identity, messaging, or governance family truth.
7. Enterprise terminology must not be used to create a parallel white-label mode.

## 10. Deferred To Later Units

This normalization deliberately does not solve:

- tenant back-office family design
- platform control-plane replanning
- domain / tenant routing / brand-surface management family design
- B2C broader family design
- catalog / discovery cross-mode design
- orders / checkout / post-purchase family design
- payments / escrow / settlement family design
- identity / tenancy / permissions / workspace family design
- messaging / notifications family design
- implementation planning or runtime changes

Those remain later bounded units.

- adjacent-family boundaries are explicit enough that later units can inherit them without
