## 6. Bundling-Boundary Rules

- If multiple labels in one surface jointly teach peer-family, separate-office, downstream-ownership,
  or taxonomy-transfer drift, they must be bundled into one future implementation-facing queue item.
- A queue item may be isolated only when its relabeling cannot leave an adjacent contradiction behind
  in the same heading, board, or grouped navigation surface.
- Aggregator heading, operations, finance, and communication labels must be treated as one office-
  implication cluster because changing only one or two still leaves Aggregator reading like a full
  execution owner.
- Discovery-safe supplier, buyer, catalog, and profile labels must be treated as one taxonomy-
  transfer-prevention cluster because mixed discovery-safe and ownership-transferring language would
  create a new contradiction.
- Composite headings that fuse family, plan, and overlay truth must be split as one bundle rather
  than patched piecemeal.

## 7. Allowed and Disallowed Queue Interpretations

Allowed:

- queue items may bundle several adjacent labels into one future implementation-facing unit when
  partial cleanup would leave the same contradiction visible
- one isolated first-wave candidate may exist where the relabel is local and does not depend on a
  larger heading or office framing change
- a future queue item may use heading split, badge add, sublabel add, rename, or group reframing as
  long as the normalized target labels are already fixed by product truth
- descriptive source surfaces may remain the evidence base for queue authoring without regaining
  canonical authority

Disallowed:

- treating queue order as implementation approval or as permission to mutate runtime in this pass
- isolating one label from a contradiction cluster when adjacent labels in the same surface would
  still teach peer-family, separate-office, downstream-ownership, or taxonomy-transfer drift
- reusing grouped plan labels, WL labels, or Aggregator overclaim labels as if they were canonical
  targets
- widening the queue into broader navigation redesign, route rewrites, or shell rewrites

## 8. Blockers and Edge Cases

- descriptive dashboard artifacts still self-describe as canonical or authoritative in places, so
  later implementation-facing work must not treat queue order as a reason to restore those
  historical documents above the six controlling artifacts and this queue
- current shell names remain code-facing continuity anchors, which means many future queue items will
  resolve through visible-label splits, badges, and sublabels rather than code identifier renames
- White Label Co remains a preserved Layer 0 hold, but it does not materially block truthful queue
  authoring in this pass
