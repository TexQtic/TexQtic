## 5. Provenance / issuance seam inspected

The exact provenance or issuance seam inspected in this pass is:

1. `components/Onboarding/OnboardingFlow.tsx` carries an `inviteToken` into the activation form
   state, but it does not create or issue that token.
2. `services/tenantService.ts` exposes `activateTenant()` as the dedicated client seam into
   `POST /api/tenant/activate`, which consumes the invite token but does not create it.
3. `server/src/routes/tenant.ts` shows that `POST /api/tenant/activate` hashes the submitted
   invite token, looks up an `invite` record by `tokenHash`, requires it to be unaccepted and
   unexpired, and then consumes that shared `invite` record through activation.
4. the same route file shows the explicit in-scope token-issuance surface: `POST /api/tenant/memberships`
   generates a random token, hashes it, stores the hash in the `invite` table with tenant, email,
   role, and expiry fields, and returns the raw token for delivery.

   adjacent guardrail that canonical provisioning and onboarding outcome or activation surfaces are
   live elsewhere, but those surfaces were not inspected here as primary issuance targets.

## 6. Repo-truth findings

The repo-truth findings from the inspected provenance seam are:

1. the exact surface that creates or issues an invite token in the inspected code is
   `POST /api/tenant/memberships` in `server/src/routes/tenant.ts`, exposed client-side through
   `createMembership()` in `services/tenantService.ts`.
2. the token consumed by the non-canonical activation path is a token backed by the shared
   `invite` table. `POST /api/tenant/activate` does not consume a separate onboarding-token type;
   it hashes the submitted token and looks up a generic `invite` record by `tokenHash`.
3. the inspected activation path therefore proves provenance only up to the shared `invite`
   primitive. Within the bounded seam, repo truth shows that the activation path consumes invite
   records from the same invite store used by the broader tenant member-invite workflow.
4. the in-scope issuance surface that is directly proven is not a distinct non-canonical
   first-owner onboarding issuance path. It is the broader tenant-plane member-invite primitive,
   which creates invite records with email, role, expiry, and token hash.
5. the inspected route-level evidence does not establish a distinct bounded provenance path showing
   that the specific invite token used by the non-canonical onboarding activation route originates
   from a separate issuance workflow dedicated to first-owner fallback.
6. the minimum adjacent guardrail evidence shows canonical provisioning and onboarding outcome or
   activation surfaces exist elsewhere, but this pass did not inspect those upstream control-plane
   surfaces directly, so it does not claim they issue the inspected invite token.
7. the truthful provenance boundary established by current repo truth is therefore limited: the
   activation seam consumes a shared `invite` token, and the only clearly inspected issuance seam is
   the broader tenant invite primitive.
