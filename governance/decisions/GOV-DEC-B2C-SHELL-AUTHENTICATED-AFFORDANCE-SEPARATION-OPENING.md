## Problem Statement

TexQtic now needs to determine whether the recovered exact B2C child should be converted into one
lawful bounded implementation opening and, if so, how to do that without widening into public-shell
work, downstream orders-family continuity, adjacent-family redesign, or prior seam re-openings.

The opening must create exactly one implementation-ready B2C child unit, preserve the already-
recorded public-facing runtime requirements, keep prior seam closures closed and separate, and avoid
opening broader B2C redesign by implication.

## Why This Opening Is Lawful Now

This opening is lawful now because the required pre-opening chain is complete and still current:

- shared public-facing runtime requirements were preserved explicitly before further B2C movement
- the broad B2C family remainder was reduced to the public-entry-to-authenticated boundary only
- that boundary was reduced again to one exact shell-level child
- no required opening-boundary question remains ambient after the shell-boundary reduction result
- current runtime anchors still exhibit the exact mixed shell condition that the child targets

The lagging `-v2` sequencing stack does not block this opening.

- `TEXQTIC-NEXT-DELIVERY-PLAN-v2.md` and `TEXQTIC-GAP-REGISTER-v2.md` still preserve B2C at
  broader family resolution only
- those `-v2` surfaces do not name a conflicting exact child, an already-open competing unit, or a
  contradictory blocker against this narrower opening
- the newer B2C decision chain supplies the fresh bounded decision that current Layer 0 requires
  before a new product-facing opening can lawfully occur

## Decision

The broader B2C family remainder remains non-open.

Exactly one bounded implementation unit is now opened:

- `MODE-COMPLETENESS-B2C-SHELL-AUTHENTICATED-AFFORDANCE-SEPARATION`
- title: `B2C shell authenticated-affordance separation`

Layer 0 should now show:

- this child as the only `OPEN` product-facing `ACTIVE_DELIVERY` unit
- no implicit second opening or successor opening beyond this child
- the broader B2C family still non-open and still separate from public-entry seam closures and
  downstream orders-family continuity

## Exact Implementation Scope Of The New Bounded Unit

The exact in-scope boundary of the child slice is:

- separate authenticated-only shell affordance exposure from the exact non-WL B2C `HOME` path
- preserve the shared branded entry-facing frame, home return, and browse-entry search continuity
  already fixed as public-facing runtime requirements
- remove or reposition authenticated-only shell affordances currently co-resident on that exact
  path, including `Orders`, `Cart`, `DPP Passport`, `Escrow`, `Escalations`, `Settlement`,
  `Certifications`, `Traceability`, `Audit Log`, `Trades`, and `Team`
- verify that the exact non-WL B2C `HOME` path no longer implies authenticated shell continuity by
  affordance presence alone

## Exact Exclusions

The exact out-of-scope boundary of the child slice is:

- shared public shell or navbar redesign
- browse-entry search redesign
- public B2C browse-entry seam reopening
- seller/admin affordance reopening
- settings affordance reopening
- orders, cart, checkout, payment, fulfillment, or post-purchase continuity changes
- downstream behavior redesign for `DPP Passport`, `Escrow`, `Escalations`, `Settlement`,
  `Certifications`, `Traceability`, `Audit Log`, `Trades`, or `Team`
- white-label, enterprise, control-plane, or Aggregator work
- onboarding, auth, domain-routing, or `g026` work
- broad B2C redesign or broader shell re-architecture

## Exact Boundary Locks Preserved

### 1. Public-Entry Continuity Lock

- preserve the branded B2C entry-facing frame
- preserve home return on the exact reviewed path
- preserve browse-entry search continuity on the exact reviewed path

### 2. Authenticated-Affordance Exposure Lock

- only authenticated-only shell affordance exposure on the exact `HOME` path is in scope
- downstream authenticated destinations remain separate from this opening
- no hidden claim of full authenticated continuity is authorized

### 3. Downstream Family Lock

- orders/cart/checkout continuity remains separate
- adjacent authenticated family behavior remains separate
- this opening controls exposure on the shell path only, not family internals

### 4. Prior-Closure Lock

- the closed public browse-entry seam remains closed
- the closed seller/admin affordance separation remains closed
- the closed settings affordance separation remains closed

### 5. Platform Lock

- no onboarding redesign
- no auth redesign
- no routing or domain work
- no `g026` reopening
