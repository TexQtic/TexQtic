# GOV-DEC-LAUNCH-ACCELERATION-OVERLAY-RESHAPE-DESIGN-OPENING

Decision ID: GOV-DEC-LAUNCH-ACCELERATION-OVERLAY-RESHAPE-DESIGN-OPENING
Title: Open one bounded concurrent governance design unit for thin launch-acceleration overlay reshape
Status: DECIDED
Date: 2026-04-05
Authorized by: Paresh

## Context

TexQtic's current authoritative governance state records that:

- no current product-facing `ACTIVE_DELIVERY` unit is open
- `NEXT-ACTION.md` and `SNAPSHOT.md` preserve the investigation result
  `PLANNING_STACK_NEEDS_RESHAPING`
- the preserved result already fixes the governing meaning:
  - the underlying need is valid for a thin launch-acceleration layer above governance
  - the currently framed broad `Launch Acceleration Planning Stack` is too wide
  - Governance OS is not the main problem and must not be expanded into broad portfolio planning
  - reshape is required before any broader planning opening may be considered
- live product-sequencing authority remains `docs/product-truth/TEXQTIC-NEXT-DELIVERY-PLAN-v2.md`
- candidate/family truth remains `docs/product-truth/TEXQTIC-GAP-REGISTER-v2.md`
- derived planning context remains `docs/product-truth/TEXQTIC-IMPLEMENTATION-ROADMAP-v2.md`

The preserved investigation already fixes the only lawful design direction for the next move:

- shape a thin `Launch Acceleration Overlay`, not a broad planning stack
- keep the concept bounded to exactly three operational surfaces only:
  - `Launch Critical Path Register`
  - `Next-Opening Shortlist Matrix`
  - `Rolling Launch Window Note`
- include one embedded stagnation rule only
- include one explicit non-duplication clause only
- keep the overlay above governance for visibility only, not above governance for authority
- preserve that lawful opening mechanics remain primary
- allow at most one narrow bridge rule only if later design proves it necessary

The preserved exclusions are already clear and must remain zero-ambiguity exclusions:

- Governance OS replacement
- doctrine-wide expansion
- roadmap regeneration
- detailed implementation planning
- new product delivery opening authority
- parallel candidate/state governance systems
- standalone verification-evidence standards
- broad family redesign
- any implementation, coding, migration, testing, or deployment activity

## Problem Statement

TexQtic now has one bounded lawful next governance move: define whether the thin overlay should be
adopted, and define its exact bounded shape, without turning the preserved reshape result into a
broad planning-stack validation or a new planning subsystem.

If this reshape move is not opened, the preserved result remains true but inactive.
If it is opened too broadly, it could be misread as validating the broader stack framing,
creating new planning authority, or authorizing product-facing implementation by implication.

The smallest truthful opening is therefore one concurrent governance-only design unit that shapes
the thin overlay only and nothing broader.

## Exact Bounded Unit Name Proposal

TexQtic opens exactly one bounded concurrent governance-only design unit:

- `LAUNCH-ACCELERATION-OVERLAY-RESHAPE-DESIGN-001`
- title: `Thin launch-acceleration overlay reshape design`

## Scope Statement

`LAUNCH-ACCELERATION-OVERLAY-RESHAPE-DESIGN-001` defines only whether TexQtic should adopt a thin
Launch Acceleration Overlay composed of the `Launch Critical Path Register`, the
`Next-Opening Shortlist Matrix`, and the `Rolling Launch Window Note`, together with one embedded
stagnation rule and one explicit non-duplication clause; it does not define a broad planning
stack, does not define implementation plans or live forecasts as operating truth, and does not sit
above Governance OS for authority, only above it for bounded visibility if later adopted.

## Entry Criteria

The unit is lawful to open only because all of the following are already true in current repo
truth:

1. the preserved investigation outcome is exactly `PLANNING_STACK_NEEDS_RESHAPING`
2. repo truth already establishes that reshape is required before any broader planning opening
3. the thin overlay direction is already bounded to the three approved operational surfaces only
4. the required exclusions are explicit and unambiguous
5. no current product-facing `ACTIVE_DELIVERY` unit is open, so this design move creates no
   conflict with an already-authorized product implementation stream

## Exit Criteria

This design-only unit completes only when all of the following are explicitly recorded:

1. the thin overlay shape is explicitly defined
2. the artifact set is bounded to the three approved operational surfaces only
3. one embedded stagnation rule is defined and bounded inside the overlay only
4. one explicit non-duplication clause is defined and preserves Governance OS as the primary
   authority for lawful openings, verification, sync, and closure
5. no new opening authority is granted to the overlay or any of its artifacts
6. any bridge rule is either rejected entirely or narrowed to one proven-necessary bridge rule only
7. the result explicitly states either:
   - a later thin-overlay opening is lawful as a separate step, or
   - the idea remains rejected or requires further refinement before any opening

## Decision

TexQtic opens exactly one bounded concurrent governance-only design unit:

- `LAUNCH-ACCELERATION-OVERLAY-RESHAPE-DESIGN-001`
- title: `Thin launch-acceleration overlay reshape design`

This unit is `OPEN` in Layer 0 as a non-terminal governance unit with delivery class
`DECISION_QUEUE`.

No product-facing unit is opened by this decision.
No broad `Launch Acceleration Planning Stack` is validated or opened by this decision.
No thin-overlay implementation or operating-truth rollout is opened by this decision.

## Delivery-Class Decision For This Open Governance Unit

`LAUNCH-ACCELERATION-OVERLAY-RESHAPE-DESIGN-001` carries delivery class `DECISION_QUEUE`.

Reason:

- the unit is governance-only design shaping, not product delivery
- the unit exists to define a possible bounded overlay only

  separately governed
- the open unit must not be mistaken for a new `ACTIVE_DELIVERY` or a new authority surface

## Exact In-Scope Boundary

The opened unit is limited to the following design-only scope only:

1. define whether TexQtic should adopt a thin `Launch Acceleration Overlay`
2. define the exact boundary of the three approved operational surfaces only:
   - `Launch Critical Path Register`
   - `Next-Opening Shortlist Matrix`
   - `Rolling Launch Window Note`
3. define one embedded stagnation rule only
4. define one explicit non-duplication clause only
5. bind any later overlay concept to current authorities:
   - Layer 0 control-plane truth
   - `docs/product-truth/TEXQTIC-NEXT-DELIVERY-PLAN-v2.md`
   - `docs/product-truth/TEXQTIC-GAP-REGISTER-v2.md`

6. preserve that the overlay is visibility above governance only, not authority above governance
7. determine whether one narrow bridge rule is necessary and preserve no bridge rule unless later
   design proves it necessary
8. define only the acceptance boundary for a later separate overlay-opening or further-refinement
   decision
