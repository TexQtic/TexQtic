# GOV-DEC-LAUNCH-ACCELERATION-OVERLAY-OPENING

Decision ID: GOV-DEC-LAUNCH-ACCELERATION-OVERLAY-OPENING
Title: Open one bounded concurrent governance visibility unit for the thin launch-acceleration overlay
Status: DECIDED
Date: 2026-04-05
Authorized by: Paresh

## Context

TexQtic's current authoritative governance state records that:

- no current product-facing `ACTIVE_DELIVERY` unit is open
- `LAUNCH-ACCELERATION-OVERLAY-RESHAPE-DESIGN-001` remains the source of overlay design truth
- the source design result is exactly `THIN_OVERLAY_DESIGN_COMPLETE_AND_LATER_OPENING_POSSIBLE`
- the source design result is a design conclusion only and does not itself authorize an overlay opening
- live product-sequencing authority remains `docs/product-truth/TEXQTIC-NEXT-DELIVERY-PLAN-v2.md`
- candidate/family truth remains `docs/product-truth/TEXQTIC-GAP-REGISTER-v2.md`
- derived planning context remains `docs/product-truth/TEXQTIC-IMPLEMENTATION-ROADMAP-v2.md`
- current product-truth still preserves no exact bounded next candidate and still requires repeated cross-file reconstruction of launch-critical sequencing, contender comparison, and window assumptions
- older launch-overlay baselines and the launch-family-chain baseline remain useful context, but they do not already provide one current bounded `Launch Critical Path Register`, one bounded `Next-Opening Shortlist Matrix`, or one bounded `Rolling Launch Window Note`

## Current-Truth Retest

The later-opening criteria recorded in `LAUNCH-ACCELERATION-OVERLAY-RESHAPE-DESIGN-001` were retested against current repo truth.

Result:

1. the overlay can still remain limited to the three approved surfaces plus one embedded stagnation rule and one explicit non-duplication clause only
2. the overlay can still derive from current authorities without competing with them
3. no extra artifact family or parallel state system is required
4. the visibility gap remains materially real because current repo truth still preserves broad contenders but no exact bounded next candidate
5. no broader bridge rule is required

## Decision-Test Summary

### Visibility-gap test

PASS.

Current Layer 0 plus live product-truth authorities still require repeated manual reconstruction to answer three current launch questions quickly:

- what is actually on the current launch-critical path
- which bounded contender is strongest for the next lawful opening
- what launch window range remains credible under current assumptions

The current launch-family-chain baseline confirms ongoing sequencing friction, but it remains a broader baseline document rather than these three bounded current-state visibility surfaces.

### Non-authority test

PASS.

The overlay remains lawful only if it stays visibility-only. This remains achievable because the source design truth already fixes:

- Layer 0 and live product-truth files remain primary
- the overlay may not open units
- the overlay may not normalize candidate state
- the overlay may not replace the roadmap or produce commitment authority

### Maintenance-burden test

PASS.

The bounded maintenance cost remains lower than the current repeated selection drag because the overlay is capped to:

- at most seven active critical-path rows
- at most five shortlist contenders
- one rolling launch-window note only
- refresh only after product-facing close, material planning-truth change, or the embedded stagnation rule firing

### Non-duplication test

PASS.

Current repo truth already has Layer 0 posture, family/candidate truth, derived roadmap context, historical launch baselines, and current family-chain baseline context. It does not already have the exact three bounded current-state visibility surfaces defined in the source design unit. A thin overlay therefore remains additive only if it summarizes those authorities rather than competing with them.

### Thinness test

PASS.

The concept still remains strictly limited to:

- `Launch Critical Path Register`
- `Next-Opening Shortlist Matrix`
- `Rolling Launch Window Note`
- one embedded stagnation rule
- one explicit non-duplication clause

No additional artifact family, queue, or bridge layer is required.

## Problem Statement

TexQtic now has a lawful basis to open the thin overlay itself because the source design is complete, the current visibility gap remains real, and the bounded overlay still avoids authority drift and duplication.

If the overlay is not opened now, the repo remains forced to reconstruct launch-critical sequencing and contender comparison manually across multiple broader authorities. If it is opened too broadly, it would regress back toward the rejected planning-stack shape.

The smallest lawful move is therefore one concurrent governance/planning visibility-only unit and nothing broader.

## Exact Bounded Unit Name Proposal

TexQtic opens exactly one bounded concurrent governance/planning visibility-only unit:

- `LAUNCH-ACCELERATION-OVERLAY-001`
- title: `Thin launch-acceleration visibility overlay`

## Scope Statement

`LAUNCH-ACCELERATION-OVERLAY-001` may maintain only the `Launch Critical Path Register`, the `Next-Opening Shortlist Matrix`, and the `Rolling Launch Window Note`, together with one embedded stagnation rule and one explicit non-duplication clause, all derived from current authorities only. It does not create product opening authority, roadmap authority, candidate-state authority, implementation-planning authority, or forecast-commitment authority.

## Entry Criteria

The unit is lawful to open only because all of the following are true in current repo truth:

1. zero product-facing `ACTIVE_DELIVERY` remains current
2. `LAUNCH-ACCELERATION-OVERLAY-RESHAPE-DESIGN-001` remains current and usable as the source of overlay design truth
3. the later-opening criteria recorded in the source design unit still pass against current repo truth
4. the overlay remains thinner than the current decision drag it is intended to reduce
5. opening the overlay does not require any widening beyond the three approved surfaces plus one rule and one clause

## Decision

TexQtic opens exactly one bounded concurrent governance/planning visibility-only unit:

- `LAUNCH-ACCELERATION-OVERLAY-001`
- title: `Thin launch-acceleration visibility overlay`

This unit is `OPEN` in Layer 0 as a non-terminal governance unit with delivery class `DECISION_QUEUE`.

No product-facing unit is opened by this decision.
No broad `Launch Acceleration Planning Stack` is validated or opened by this decision.
No thin overlay authority is created beyond the bounded visibility unit recorded here.

## Delivery-Class Decision For This Open Governance Unit

`LAUNCH-ACCELERATION-OVERLAY-001` carries delivery class `DECISION_QUEUE`.

Reason:

- the unit is governance/planning visibility-only, not product delivery
- it exists to keep the current critical path, shortlist, and launch window visible in bounded form only
- any later product-facing opening still requires a separate bounded governance move
- the unit must not be mistaken for a new `ACTIVE_DELIVERY` or a new authority layer

## Exact In-Scope Boundary

The opened unit is limited to the following visibility-only scope only:

1. maintain one bounded `Launch Critical Path Register`
2. maintain one bounded `Next-Opening Shortlist Matrix`
3. maintain one bounded `Rolling Launch Window Note`
4. apply one embedded stagnation rule only
5. preserve one explicit non-duplication clause only
6. derive overlay content from current Layer 0 and live product-truth authorities only
7. preserve that `LAUNCH-ACCELERATION-OVERLAY-RESHAPE-DESIGN-001` remains the source of overlay design truth

## Exact Out-of-Scope Boundary

This decision does **not** authorize:

- Governance OS replacement
- doctrine-wide expansion
- roadmap regeneration
- detailed implementation planning
- any new product delivery opening authority
- any candidate-state normalization authority
- any forecast-commitment authority
- any parallel candidate/state system
- any second overlay or planning child opened by implication
- any application, server, package, database, schema, test, or infra change

## Implementation Authorization Statement

This decision authorizes exactly one bounded governance/planning visibility-only unit only:

- `LAUNCH-ACCELERATION-OVERLAY-001`

It does **not** authorize a broad planning stack.
It does **not** authorize a product-facing implementation unit.
It does **not** authorize the overlay to compete with Layer 0 or live product-truth authorities.
